一、 索引原理

1、 索引概述

MySQL官方对索引定义：索引是MySQL存储引擎用于快速查找记录的一种数据结构。

简单理解为： 索引是 排好序的数据结构，帮助我们快速的查询数据库表中的数据。

索引是一种特殊的数据文件 ：

MyISAM存储引擎：数据文件和索引文件是分开的。索引文件中保存的是数据记录的地址。

InnoDB存储引擎：表数据本身就是按照B+Tree组织的索引结构。.ibd文件 就是数据+索引存储文件。索引是一种数据结构

索引是一个独立的、物理的数据结构，它是由一个表中的一个字段或者多个字段的值组合成的集合。

MySQL中，默认使用的是B+Tree结构管理索引。

2、索引底层数据结构

InnoDB 底层采用 B+ 数来存储和组织数据， 其他类型 数据结构问题如下：

  • 二叉查找树
    • 在存储有序数据的时候，最终排列结构会形成一个单向链表，对于读取链表尾部的数据时，效率会很低。
  • AVL 树 / 红黑树
    • 一个节点最多分裂出两个子节点，树的高度太高，导致IO次数过多
    • 节点里面只保存一个关键字，每次操作获取的目标数据太少
  • B 树
    • B-Tree中每个节点不仅包含数据的key值，还有data数据。当data数据较大的时候，会导致每个节点存储的key值减少了，就会导致B树的层数变高。增加查询的I/O 次数。

B+ 树特点

  • 所有数据存储在叶子节点，叶子节点形成有序链表，支持范围查询和顺序访问。
  • 非叶子节点仅存储索引信息，不保存数据，查询效率稳定。
  • 树的高度较低，减少磁盘I/O操作。
  • 适用场景：
    • 范围查询（如BETWEEN、>、<）。
    • 频繁的插入和删除操作。

3、 MySQL 索引核心概念

以下是关于聚簇索引与非聚簇索引、回表查询与覆盖索引、联合索引与最左前缀法则的总结归纳：

聚簇索引与非聚簇索引

  • 聚簇索引（Clustered Index）
    • 定义：聚簇索引决定了数据在物理存储上的顺序。表中的数据行存储顺序与索引的顺序一致。
    • 特点：
      • 每张表只能有一个聚簇索引。
      • 数据行存储在索引的叶子节点中，因此查询效率高，尤其是范围查询。
      • 插入、更新和删除操作可能较慢，因为数据行的物理位置可能需要调整。
      • 适用于主键或唯一性约束的字段。
    • 适用场景：适合频繁查询的字段，尤其是范围查询（如 SELECT * FROM table WHERE id BETWEEN 100 AND 200）。
  • 非聚簇索引（Non-Clustered Index）
    • 定义：非聚簇索引与数据行的存储顺序无关。索引存储的是指向数据行的指针。
    • 特点：
      • 一张表可以有多个非聚簇索引。
      • 索引的叶子节点存储的是指向数据行的指针，而非数据本身。
      • 查询时可能需要回表查询（即先通过索引找到指针，再通过指针找到数据行）。
      • 对数据行的物理存储顺序无影响，因此插入、更新和删除操作相对高效。
    • 适用场景：适合用于精确查询的字段（如 SELECT * FROM table WHERE column = 'value'），尤其是当查询字段不频繁用于范围查询时。

回表查询与覆盖索引

  • 回表查询（Index Backing）
    • 定义：当查询需要的数据不在索引中时，数据库引擎会先通过索引找到数据行的指针，再通过指针访问实际的数据表，获取所需数据。
    • 特点：
      • 额外的I/O操作，性能开销较大。
      • 通常发生在非聚簇索引查询中，因为非聚簇索引的叶子节点不包含完整的数据行。
    • 优化建议：尽量减少回表查询，可以通过优化索引设计或调整查询语句来避免。
  • 覆盖索引（Covering Index）
    • 定义：如果查询所需的所有字段都包含在索引中，数据库可以直接通过索引获取数据，而无需回表查询。
    • 特点：
      • 提高查询效率，减少I/O操作。
      • 索引的叶子节点包含查询所需的所有字段。
    • 适用场景：适用于查询字段较少且可以通过索引完全覆盖的场景（如 SELECT column1, column2 FROM table WHERE column1 = 'value'，其中 column1 和 column2 都在索引中）。

联合索引与最左前缀法则

  • 联合索引（Composite Index）
    • 定义：联合索引是基于多个字段创建的索引，索引的顺序按照字段的定义顺序排列。
    • 特点：
      • 提高多字段查询的效率。
      • 索引的顺序很重要，查询时必须按照索引的顺序使用字段。
      • 可以部分使用联合索引（即使用索引的前缀字段）。
    • 适用场景：适用于多字段联合查询（如 SELECT * FROM table WHERE column1 = 'value1' AND column2 = 'value2'）。
  • 最左前缀法则（Left-Most Prefix Rule）
    • 定义：在联合索引中，查询条件必须从索引的最左边字段开始匹配，才能有效利用索引。
    • 特点：
      • 如果查询条件跳过了索引的最左边字段，则索引可能无法被有效利用。
      • 索引的字段顺序对查询性能影响很大。
    • 示例：
      • 索引为 (column1, column2, column3)：
        • 查询条件为 WHERE column1 = 'value1' AND column2 = 'value2' 可以有效利用索引。
        • 查询条件为 WHERE column2 = 'value2' AND column3 = 'value3' 则无法有效利用索引。
    • 优化建议：在设计联合索引时，应根据查询频率和字段的过滤性合理安排字段顺序。

前缀索引

  • 定义：前缀索引是基于字符串字段的前缀创建的索引，而不是整个字段。它通常用于长字符串字段，以减少索引大小。
  • 特点：
    • 可以显著减少索引的大小，节省存储空间。
    • 但可能会降低索引的唯一性，增加查询的复杂性。
  • 应用场景：适用于长字符串字段（如用户名、URL等），尤其是当字段值较长且前缀重复性较低时。
  • 注意事项：需要合理选择前缀长度，以平衡索引大小和查询性能。

二、优化分析

1、 Explain 指令

MySQL 的 EXPLAIN 指令是优化 SQL 查询的重要工具，它可以帮助你分析查询的执行计划，理解查询是如何被优化器处理的。以下是 EXPLAIN 的详细梳理：

基本用法

在 SQL 查询前添加 EXPLAIN 或 EXPLAIN FORMAT=JSON（更详细）即可查看执行计划：

输出字段解析

EXPLAIN 输出的关键字段及其含义：

1. id

  • 作用：查询中 SELECT 子句的执行顺序。
  • 规则：
    • 相同 id 表示同一执行层级（如普通查询）。
    • id 递增表示嵌套查询（如子查询），值越大优先级越高。

2. select_type

  • 作用：查询类型。
  • 常见值：
    • SIMPLE：简单查询（无子查询或 UNION）。
    • PRIMARY：外层主查询。
    • SUBQUERY：子查询中的第一个 SELECT。
    • DERIVED：派生表（FROM 子句中的子查询）。
    • UNION：UNION 中的第二个或后续查询。
    • UNION RESULT：UNION 的结果。

3. table

  • 作用：当前行操作的表名（可能是别名或派生表名如 <derivedN>）。

4. partitions

  • 作用：查询匹配到的分区（若表未分区则为 NULL）。

5. type

  • 作用：访问数据的方式（性能从优到劣排序）：
    • system：表只有一行（系统表）。
    • const：通过主键或唯一索引直接找到一行（如 WHERE id = 1）。
    • eq_ref：唯一索引关联（如 JOIN 中使用主键）。
    • ref：非唯一索引查找（返回多个行）。
    • range：索引范围扫描（如 BETWEEN、IN）。
    • index：全索引扫描（遍历索引树）。
    • ALL：全表扫描（需优化）。

6. possible_keys

  • 作用：可能用到的索引（实际可能未使用）。

7. key

  • 作用：实际使用的索引（若为 NULL 表示未用索引）。

8. key_len

  • 作用：使用的索引长度（单位字节），可用于判断联合索引的使用情况。

9. ref

  • 作用：显示索引的哪一列被使用（如 const 或列名）。

10. rows

  • 作用：预估需要扫描的行数（越小越好）。

11. filtered

  • 作用：查询条件过滤后剩余行的百分比（仅限 MySQL 5.7+）。

12. Extra

  • 作用：额外信息（常见值）：
    • Using index：覆盖索引（无需回表）。
    • Using where：WHERE 过滤行。
    • Using temporary：使用临时表（常见于排序或分组）。
    • Using filesort：需要额外排序（需优化）。
    • Using join buffer：使用连接缓存（嵌套循环连接）。

优化建议

通过 EXPLAIN 分析后，可采取以下优化策略：

  1. 避免全表扫描（type=ALL）：为 WHERE 条件中的列添加索引。
  2. 优化索引使用：
    • 确保索引覆盖查询字段（减少回表）。
    • 避免索引失效（如对索引列使用函数、类型转换）。
  3. 减少 rows 值：通过更精确的查询条件减少扫描行数。
  4. **消除 Using filesort 或 Using temporary**：优化 ORDER BY/GROUP BY，使用索引排序。
  5. 关注 Extra 字段：优先解决 Using filesort 或 Using temporary。

示例分析

假设输出：

  • type=ref（使用非唯一索引 user_id）
  • key=user_id
  • Extra=Using filesort（需要额外排序）

优化：为 (user_id, order_date) 创建联合索引，消除文件排序。

注意事项

  • EXPLAIN 结果是基于统计信息的估算，可能与实际执行有差异。
  • 使用 EXPLAIN ANALYZE（MySQL 8.0+）可获取实际执行数据。
  • 分区表、子查询和复杂 JOIN 可能需要更深入分析。

通过 EXPLAIN，可以快速定位查询瓶颈并针对性优化，是 SQL 性能调优的必备工具。

3、 主键选型

选择合适的主键是数据库设计中的一个重要环节，它直接影响到数据的完整性、查询性能和系统的可扩展性。以下是一些关于主键选择的建议：

理解主键的作用

主键的主要作用是唯一标识表中的每一行数据。它必须满足以下条件：

  • 唯一性：主键值不能重复。
  • 非空性：主键值不能为空（NOT NULL）。
  • 稳定性：主键值在表的生命周期内不应更改。

常见的主键类型及优缺点

（1）自然主键（Natural Key）

  • 定义：使用数据本身的属性作为主键，如身份证号、用户名、邮箱等。
  • 优点：
    • 语义化强，易于理解和维护。
    • 减少冗余，避免额外的字段存储。
  • 缺点：
    • 可能不够稳定（如用户可能更换邮箱）。
    • 可能导致索引膨胀（如身份证号等长字符串）。
    • 查询性能可能较低（尤其是长字符串作为主键）。
  • 适用场景：当自然键稳定且唯一性有保障时，如银行账户号、商品条码等。

（2）代理主键（Surrogate Key）

  • 定义：使用数据库自动生成的值作为主键，通常是整数（如自增ID）。
  • 优点：
    • 稳定且唯一，不会因业务逻辑变化而改变。
    • 索引效率高（整数占用空间小，查询速度快）。
    • 与业务逻辑解耦，便于扩展。
  • 缺点：
    • 缺乏语义化，需要额外字段存储业务标识。
    • 需要额外维护外键关系。
  • 适用场景：大多数需要高性能和高扩展性的场景，尤其是业务逻辑复杂或数据频繁更新的系统。

主键选择的具体建议

（1）优先选择代理主键（自增ID）

  • 理由：

    • 性能优势：整数类型的自增ID占用空间小，索引效率高，适合频繁的增删改查操作。
    • 稳定性：自增ID不会因业务逻辑变化而更改，适合需要高扩展性的系统。
    • 通用性：适用于大多数场景，尤其是业务逻辑复杂或数据量较大的系统。

（2）使用自然主键时需谨慎

  • 适用条件：

    • 自然主键必须是唯一且稳定的，且不会因业务逻辑变化而更改。
    • 自然主键的字段类型应尽量短（如短字符串或数字）。

  • 示例：

    • 适用场景：商品表中使用商品条码（如SKU）作为主键。

（3）避免使用复合主键

  • 理由：

    • 复合主键（联合主键）由多个字段组成，增加了索引的复杂性和查询成本。
    • 复合主键的唯一性依赖于多个字段，维护成本较高。

  • 替代方案：使用代理主键，并通过唯一约束（UNIQUE）来保证字段组合的唯一性。

（4）考虑业务需求和数据特性

  • 唯一性：确保主键字段在业务逻辑中是唯一的。
  • 稳定性：主键字段不应因业务逻辑变化而更改。
  • 性能：尽量选择短字段（如整数或短字符串）作为主键，避免长字符串（如UUID）。
  • 扩展性：如果系统可能扩展到分布式架构，考虑使用全局唯一标识符（如UUID）作为主键，但需注意性能影响。

（5）特殊场景的主键选择

  • 分布式系统：如果系统需要分布式部署，建议使用全局唯一标识符（如UUID或自定义的分布式ID生成策略）。

    • 优点：避免主键冲突，支持分布式扩展。
    • 缺点：UUID长度较长，索引性能略低于整数。

总结

  • 优先选择代理主键（自增ID），因为它简单、高效且稳定。
  • 谨慎使用自然主键，确保其唯一性和稳定性。
  • 避免使用复合主键，改用唯一约束来保证字段组合的唯一性。
  • 根据业务需求和数据特性选择合适的主键类型，并在分布式系统中考虑全局唯一标识符。
  • 性能优先：在大多数情况下，主键的性能和稳定性比语义化更重要。

通过合理选择主键，可以提高数据库的性能、可维护性和可扩展性。

4、索引失效问题梳理

查询条件问题

  • 使用函数或表达式 —— 在索引列上使用函数（如 YEAR()、LEFT()）或表达式会导致索引失效。
    • 示例：SELECT * FROM employees WHERE YEAR(birthdate) = 1990;
    • 原因：数据库无法直接对函数结果使用索引。
    • 优化：改写为范围查询，如 birthdate >= '1990-01-01' AND birthdate < '1991-01-01'。
  • 使用 OR 条件 —— 如果 OR 连接的条件中有一个未使用索引，可能会导致索引失效。
    • 示例：SELECT * FROM employees WHERE age = 30 OR name = 'John';
    • 优化：使用 UNION 替代 OR，分别查询后合并结果。
  • 使用 LIKE 通配符开头 —— 当 LIKE 查询以通配符（如 % 或 _）开头时，索引失效。
    • 示例：SELECT * FROM employees WHERE name LIKE '%John';
    • 优化：尽量将通配符放在查询条件的末尾。
  • 数据类型不匹配 —— 查询条件的数据类型与索引列类型不一致时，可能导致隐式类型转换，从而索引失效。
    • 示例：SELECT * FROM employees WHERE salary = '30000';（salary 是 INT 类型）
    • 优化：确保查询条件的数据类型与索引列一致。
  • 范围查询阻断 —— 在联合索引中，范围查询会阻断后续列的索引使用。
    • 示例：SELECT * FROM employees WHERE age BETWEEN 30 AND 40 AND name = 'John';
    • 优化：调整查询条件顺序，确保范围查询在最后。

索引结构问题

  • 联合索引未遵循最左前缀原则 —— 查询条件未按联合索引的列顺序使用时，索引可能失效。
    • 示例：CREATE INDEX idx_name_age ON employees (name, age);
      查询：SELECT * FROM employees WHERE age = 30 AND name = 'John';
    • 优化：调整查询条件顺序，或重新设计联合索引。
  • 索引列参与计算 —— 索引列在查询条件中参与计算时，索引失效。
    • 示例：SELECT * FROM employees WHERE age + 10 > 40;
    • 优化：改写为直接使用索引列，如 age > 30。

特殊查询语句问题

  • 使用 NOT、IS NULL 或 IS NOT NULL —— 这些操作符可能导致索引失效。
    • 示例：SELECT * FROM employees WHERE NOT age = 30;
    • 优化：尽量避免使用这些操作符，或通过其他方式重写查询。
  • 使用 DISTINCT 或 GROUP BY —— 这些操作可能导致索引无法完全利用。
    • 示例：SELECT DISTINCT name FROM employees WHERE age = 30;
    • 优化：确保查询条件能够充分利用索引。
  • 使用 LIMIT 时未明确排序 —— 如果查询包含 LIMIT 但未使用排序索引，索引可能失效。
    • 示例：SELECT * FROM employees WHERE age = 30 LIMIT 10;
    • 优化：添加 ORDER BY 条件。

连接查询问题

  • 连接条件未使用索引 —— 在 JOIN 查询中，连接条件（ON 子句）未涉及索引时，索引失效。
    • 示例：SELECT * FROM employees e JOIN departments d ON e.department_id = d.id;
    • 优化：确保连接条件涉及的列已建立索引。

其他问题

  • 查询条件未使用索引列 —— 查询条件未涉及索引列时，数据库会直接执行全表扫描。
    • 优化：为需要频繁查询的列创建索引。
  • 隐式类型转换 —— 查询中对列进行隐式类型转换，可能导致索引失效。
    • 优化：确保查询条件的数据类型与索引列一致。
  • 字符串拼接或连接运算 —— 对索引列进行拼接或连接运算时，索引失效。
    • 示例：SELECT * FROM employees WHERE CONCAT(first_name, ' ', last_name) = 'John Doe';
    • 优化：避免对索引列进行复杂运算。

1.5 JOIN优化

JOIN使用总结：

  1. 永远用小的结果集去驱动大的结果集（本质就是减少外层循环的数据数量）。
  2. 应该为匹配的条件 增加索引（减少内层表的循环匹配次数）。
  3. 增加 join buffer的大小（一次缓存的越多，内层表扫描的次数就越少）
  4. 减少不必要的字段查询（字段越少，join buffe所缓存的数据就越多)。

1.7 MySQL查询成本计算

1.7.1 什么是成本？

在MySQL中，一条查询语句的执行成本实际上是由两部分构成的：

  • I/O 成本

  • MyISAM 和 InnoDB都需要将数据和素银存储到磁盘，当进行查询时，就需要把数据或者索引加载到内存中。从磁盘到内存这个加载过程，损耗的时间，我们称之为 I/O成本。

  • CPU成本

  • 读取记录的时候，需要检测记录是否满足对应的 搜索条件、对结果集进行排序等等操作发生的损耗都称之为 CPU成本。

  • 成本常数

  • I/O成本常数：1.0 ，MySQL规定读取一个页面花费的成本默认是1.0

  • CPU成本常数：0.2，读取以及检测一条记录是否符合搜索条件的成本，默认是0.2

1.7.2 单表查询成本

MySQL中找出成本最低方案的过程，大致如下

  1. 根据搜索条件，找出所有可能使用的索引。
  2. 计算全表扫描的代价。
  3. 计算使用不同索引，执行查询的代价。
  4. 对比各种执行方案的代价，找出成本最低的那个。
